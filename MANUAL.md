# DocHub Plugin 使用手冊

> **DocHub** 是一套內建於 NocoBase 的企業文件管理系統，由 AI 全端工程師從零開始設計與實作。  
> 支援 Markdown 撰寫、權限管理、版本歷史、全文搜尋、站內通知、Git 雙向同步（GitHub / GitLab），以及三層組織架構。

## 1. 系統架構

DocHub 作為 NocoBase 自訂 Plugin 運行，完全整合在現有後台，不需要額外部署任何服務。

```
NocoBase v2.0.32 (Docker)
└── @nocobase/plugin-doc-hub
    ├── 前端 (AMD/UMD React，無 JSX)
    │   ├── ListPage      — 文件列表、搜尋、篩選
    │   ├── ViewPage      — 閱讀模式（Markdown 渲染）
    │   ├── EditPage      — Markdown 編輯器（分割預覽）
    │   └── VersionPage   — 版本歷史 + 逐行 Diff
    └── 後端 (Node.js + Sequelize)
        ├── 自訂 Action Handlers
        ├── afterCreate Hook（自動建 v1 版本）
        ├── afterUpdate Hook（自動建版本、發通知）
        └── PostgreSQL 資料庫（7 張表）
```

**技術亮點：**
- 純 `React.createElement` 撰寫，不需要 JSX 編譯，直接 deploy dist 即生效
- 完整 AMD/UMD 格式，符合 NocoBase 的 RequireJS 載入機制
- Markdown 渲染用本地 serve 的 `marked.min.js`，不依賴 CDN
- 支援 GitHub 與 GitLab（含自簽憑證）雙向同步

## 2. 快速開始

### 進入 DocHub

登入 NocoBase 後，點擊左側 sidebar 的 **DocHub** 連結，進入文件列表頁面。

### 介面概覽

```
┌─────────────────────────────────────────────────────────────────┐
│  Doc Hub                                          [新增文件]     │
├──────────┬──────────────────────────────────────────────────────┤
│          │  全部  SRS  SDS  SPEC  PM-Doc  Others               │
│  🔍搜尋  ├──────────────────────────────────────────────────────┤
│          │  標題   類型  狀態  最後更新      Git  操作           │
│  群組    │  ──────────────────────────────────────────────────  │

> 操作欄圖示：✏ 編輯　📋 版本歷史　⇄ 移動文件　🔄 同步 Git　🗑 刪除

## 3. 三層組織架構

DocHub 支援三層結構，讓大型團隊可以清楚分類管理文件。

```
群組（Group）          — 最高層，例：「專案」「共用」「部門文件」
  └── 專案（Project）  — 對應實際業務或 Git repo
       └── 資料夾（Category）  — 文件分類，支援無限層級巢狀
            └── 文件（Document）
```

### 建立群組（Admin 限定）

1. 在 sidebar「群組」標題右側點擊 **+**
2. 輸入群組名稱（例：`專案`、`共用`）
3. 點「建立」

### 建立專案（Admin 限定）

1. 在任一群組右側點擊 **+**
2. 輸入專案名稱
3. 點「建立」

> 只有 Admin（NocoBase root / admin 角色）才能建立、刪除群組和專案。

### 刪除專案（Admin 限定）

1. 將滑鼠移到專案名稱上，右側出現 **×** 按鈕
2. 點擊 × → 彈出二次確認視窗
3. 確認視窗顯示紅色警告，必須點「確認刪除」才會執行

### 建立資料夾

- **專案右側的 `+`** → 在該專案根目錄建立資料夾
- **資料夾右側的 `+`** → 在該資料夾底下建立子資料夾

建立時 Modal 會顯示「建立在『XXX』底下」提示確認位置。

> **注意：同一層不允許建立相同名稱的資料夾。**（系統會即時提示錯誤）

### 重新命名資料夾

**雙擊** sidebar 上的資料夾名稱 → 進入 inline 編輯模式：
- 輸入新名稱，按 **Enter** 確認 / **Escape** 取消 / 點其他地方也會儲存
- 若同層已有相同名稱，系統會提示錯誤並取消改名

### 展開 / 折疊 / 篩選

- 點擊 ▸ / ▾ 展開或折疊子資料夾
- 點擊**資料夾名稱** → 右側只顯示該分類下的文件
- 點擊**專案名稱** → 顯示整個專案的文件，同時顯示文件數量 badge
- 再點一次 → 取消篩選，顯示全部

## 4. 撰寫文件

點擊右上角「**新增文件**」或列表中的 ✏ 編輯按鈕進入編輯頁。

### 兩種建立模式

#### 模式 A：手動撰寫

1. 點「新增文件」
2. 填寫文件標題（**必填**，欄位有紅色 `*` 標示）
3. 選擇文件類型與狀態
4. 在左側 Markdown 編輯區撰寫，右側即時預覽
5. 點「儲存」或「發布」

> 從 sidebar 的資料夾點「新增文件」，會自動帶入所屬專案與資料夾，不需要手動選擇。

> **注意：同一資料夾下不允許建立相同標題的文件。**（系統會即時提示錯誤）

#### 模式 B：從 Git 匯入

1. 點「新增文件」
2. 在頁面上方綠色「Git 雙向同步」欄位填入：
   - **Repo**：`owner/repo`（GitHub）或 `10.1.2.191/namespace/project`（GitLab）
   - **檔案路徑**：`README.md` 或 `docs/spec.md`
   - **分支**：`master` 或 `main`（預設 master）
3. 點「**從 Git 拉取內容**」→ 自動填入文件內容
4. 補上標題後儲存

### 編輯器介面

```
┌─────────────────────────────────────────────────────┐
│  ← 返回  * [文件標題（必填）]    [儲存] [發布] [同步Git] │
├─────────────────────────────────────────────────────┤
│  所屬專案: [公司知識庫 ▼]  所屬資料夾: [Starters ▼]  │
│  文件類型: [SPEC ▼]        狀態: [Draft ▼]           │
├─────────────────────────────────────────────────────┤
│  🔗 Git 雙向同步                                     │
│  Repo: [owner/repo 或 10.1.2.191/ns/proj]           │
│  路徑: [README.md]  分支: [master]  [從 Git 拉取]   │
├─────────────────────────────────────────────────────┤
│  Viewers: Tom Amy  [+ 加入]                         │
│  Editors: Tom      [+ 加入]                         │
│  Subscribers: Amy  [+ 加入]                         │
├──────────────────────┬──────────────────────────────┤
│ B  I  U  H1 H2 H3   │                              │
│ — • 1. 🔗 </>       │                              │
│  [Markdown 輸入區]   │    [即時預覽]                │
└──────────────────────┴──────────────────────────────┘
```

### Markdown 工具列

| 按鈕 | 說明 |
|------|------|
| **B** | 粗體（`**text**`）|
| *I* | 斜體（`_text_`）|
| <u>U</u> | 底線（`<u>text</u>`）|
| H1 / H2 / H3 | 標題（`# ## ###`）|
| • | 無序列表（`- `）|
| 1. | 有序列表（`1. `）|
| 🔗 | 超連結（`[text](url)`）|
| `</>` | 程式碼區塊（` ``` `）|

選取文字再點按鈕 → 自動包住選取的文字。

### 儲存 vs 發布

| 動作 | 說明 |
|------|------|
| **儲存** | 存為 Draft，不對外公開 |
| **發布** | 狀態改為 Published，才能同步到 Git |

### 版本備註

儲存或發布時會彈出輸入框，可填寫這次改動了什麼（選填）。

- 填了 → 版本歷史顯示你填的說明
- 沒填 → 自動顯示版本號（`v1`, `v2`, ...）

## 5. 閱讀模式

在文件列表點擊**文件標題** → 進入閱讀模式（ViewPage）。

- Markdown 完整渲染（表格、程式碼高亮、標題錨點、blockquote）
- 顯示文件類型、最後更新時間、最後編輯者姓名
- 有編輯權限 → 右上角出現「編輯」按鈕
- 沒有查看權限 → 顯示 🔒 畫面

## 6. 權限管理

每篇文件獨立設定三種角色，在 EditPage 的 Permissions 區塊操作。

### 三種角色

| 角色 | 說明 | 提示 |
|------|------|------|
| **Viewers** | 可以看到這篇文件（否則連列表都不會出現）| 通常只有「純看」的人才需要特別加這裡；Editors / Subscribers 已自動包含讀取權限 |
| **Editors** | 可以進入編輯模式修改內容 | 自動具備讀取權限，無需同時加入 Viewers |
| **Subscribers** | 文件每次修改時自動收到站內通知 | 自動具備讀取權限，無需同時加入 Viewers |

### 設定方式

1. 進入 EditPage
2. 在 Permissions 區塊點「**加入**」按鈕
3. 從下拉選單選擇用戶
4. 儲存文件

### 存取規則（強制隔離）

| 身份 | 可看到的文件 |
|------|------------|
| **Admin**（root / admin 角色） | 所有文件，不受 Viewers 限制 |
| **文件擁有者**（建立者） | 自己建立的所有文件，自動擁有完整權限 |
| **一般用戶** | 只能看到「自己在 Viewers / Editors / Subscribers 清單」的文件 |

> **重要**：沒有設定任何 Viewers 的文件，一般用戶在列表看不到，也無法進入。

### 範例場景

```
文件：「後端部署 SOP」
Viewers:     工程師 A、工程師 B（其他人看不到）
Editors:     工程師 A（只有 A 可以改，A 同時也能讀，不需加入 Viewers）
Subscribers: 工程師 B（A 改動時 B 收通知，B 同時也能讀，不需加入 Viewers）
```

## 7. 全文搜尋

在 sidebar 頂部的搜尋框輸入關鍵字，400ms 自動觸發搜尋。

### 搜尋範圍
- 文件**標題**
- 文件**全文內容**（不分大小寫）

### 搜尋結果
- 標題命中 → 標題文字**黃底高亮**
- 內容命中 → 每筆文件顯示最多 **3 個命中片段**（前後各 60 字），黃色左邊線標示
- 片段裡的關鍵字同樣黃底高亮

### 組合篩選

搜尋可以和以下條件同時使用：
- Sidebar 的**專案**篩選
- Sidebar 的**資料夾**篩選
- 頂部的**文件類型** Tab（全部 / SRS / SDS / SPEC / PM-Doc / Others）

## 8. 版本歷史與 Diff

在文件列表點擊 📋（歷史）按鈕 → 進入版本歷史頁。

### 版本列表（左側）
- **建立文件時自動產生 v1 版本**
- 每次儲存 / 發布都自動建立新版本快照
- 顯示版本號、時間、**編輯者姓名**、版本備註

### 逐行 Diff（右側）

選擇版本後，右側顯示與上一版本的逐行差異：

```diff
  # 後端部署 SOP
  
- ## 步驟一：啟動服務
+ ## 步驟一：確認環境變數
  
+ 確認 .env 檔案存在後，再執行：
  docker compose up -d
```

- **綠色 `+`** = 這個版本新增的行
- **紅色 `-`** = 這個版本刪除的行
- 右上角顯示 `+N行 -N行` 統計
- v1（初始版本）對比「空白」，全部顯示為新增

## 9. 站內通知

文件的 **Subscribers** 在文件被修改時，自動收到 NocoBase 站內信通知。

### 通知內容
```
標題：文件更新：後端部署 SOP
內容：Tom 更新了文件《後端部署 SOP》（修正 API 路徑錯誤）
```

- 有填版本備註 → 通知裡附上備註內容
- 通知出現在 NocoBase 右上角鈴鐺，顯示紅點（WebSocket 即時推送）
- 編輯者不會通知自己

## 10. Git 雙向同步

每篇文件可以個別綁定一個 Git 檔案路徑，支援 **GitHub** 和 **GitLab**（含公司內網自簽憑證）。

### 支援的 Git 平台

| 平台 | Repo 格式 | 說明 |
|------|---------|------|
| GitHub | `owner/repo` | 例：`yanchen184/my-project` |
| GitLab（公司） | `10.1.2.191/namespace/project` | 例：`10.1.2.191/wezoomtek/wez-spring-boot-starters` |
| GitLab（帶 https）| `https://10.1.2.191/namespace/project` | 兩種格式皆支援 |

### 綁定文件與 Git 檔案

在 EditPage 頂部的綠色「🔗 Git 雙向同步」欄位：

| 欄位 | 說明 | 範例 |
|------|------|------|
| **Repo** | Git repo 路徑 | `10.1.2.191/wezoomtek/wez-spring-boot-starters` |
| **檔案路徑** | repo 內的相對路徑 | `README.md` 或 `docs/spec.md` |
| **分支** | 要同步的分支（預設 master） | `master` 或 `main` |

> Git 欄位沒有填寫時，列表和編輯頁都不會顯示同步按鈕。

### Git → DocHub（兩種方式）

#### 方式 A：手動拉取

在 EditPage 點「**從 Git 拉取最新**」→ 立即從 Git 拉取最新內容覆蓋 DocHub。

- 新增文件時按鈕顯示「**從 Git 拉取內容**」
- 已有文件時按鈕顯示「**從 Git 拉取最新**」

#### 方式 B：Webhook 自動同步（推薦）

設定後，每次 Git push 都會自動觸發 DocHub 同步，無需手動操作。

**Webhook 設定步驟（GitLab）：**

1. 進入 GitLab repo → **Settings → Webhooks**
2. URL 填入：`https://{你的公開URL}/api/docDocuments:webhookReceive`
3. Trigger 勾選 **Push events**
4. 取消勾選 **SSL verification**（若使用 ngrok 或自簽憑證）
5. 點 **Add webhook**

**公開 URL 取得方式：**
- **ngrok（臨時）**：執行 `ngrok http 13000`，取得 `https://xxxx.ngrok-free.app`
  - 注意：免費版每次重啟 URL 會變，需重新設定 Webhook
- **固定 IP（推薦）**：將 NocoBase 部署到公司內網有固定 IP 的機器

Webhook endpoint：`POST /api/docDocuments:webhookReceive`（無需登入，public）

### DocHub → Git（手動推送）

1. 文件必須先**發布**（Draft 無法同步）
2. 在列表頁點擊 🔄 按鈕，或在 EditPage 點「**同步 Git**」
3. 確認後文件內容推送到 Git 對應路徑

### 衝突處理

當 Git 已有新版本，但 DocHub 尚未同步時：

1. 在 DocHub 點儲存 → 系統偵測到 SHA 不一致
2. 彈出**衝突警告 Modal**：「此文件的 Git 版本已有更新」
3. 點「同步 Git 最新版本」→ 拉取最新內容
4. 重新編輯後再儲存

> **原則：先同步再編輯**，確保不會蓋掉工程師的更新。

### 同步狀態

| 狀態 | 說明 |
|------|------|
| `Pending` | 尚未同步過 |
| `Synced ✓` | 最後一次同步成功 |
| `Failed ✗` | 同步失敗（Token 過期或 repo 設定錯誤）|

## 11. 文件與資料夾管理

### 移動文件

將文件移動到不同的專案或資料夾：

1. 在文件列表，找到要移動的文件
2. 點擊操作欄的 **⇄ 移動** 按鈕
3. 在彈出的 Modal 中選擇**目標專案**
4. 再選擇**目標資料夾**（可選）
5. 點「確認移動」

> 移動後文件會立即出現在新位置，列表自動重新整理。

### 刪除文件

1. 在文件列表，找到要刪除的文件
2. 點擊操作欄的 **🗑 刪除** 按鈕
3. 彈出二次確認視窗，確認文件名稱
4. 點「確認刪除」執行（此操作**不可復原**）

> 任何有編輯權限的用戶都可以刪除文件。Admin 可以刪除所有文件。

### 重新命名資料夾

**雙擊** sidebar 上的資料夾名稱 → inline 編輯：
- **Enter** 確認　**Escape** 取消　點其他地方也會儲存
- 若同層已有相同名稱，系統會提示錯誤並還原

### 刪除資料夾（Admin 限定）

1. 在 sidebar 將滑鼠移到資料夾名稱上，右側出現 **×** 按鈕
2. 點擊 × → 彈出二次確認視窗
3. 點「確認刪除」執行

> **注意**：刪除資料夾後，該資料夾下的文件不會自動刪除，只是失去分類歸屬。

### 唯一名稱限制

| 範圍 | 規則 |
|------|------|
| 同一層資料夾 | 不允許建立或改名為相同名稱 |
| 同一資料夾下的文件 | 不允許建立或修改為相同標題 |

## 12. 管理員功能

Admin（NocoBase root 或 admin 角色）擁有以下額外功能：

| 功能 | 說明 |
|------|------|
| 建立群組 | Sidebar「群組」旁的 + |
| 刪除群組 | 群組旁的 × → 二次確認 |
| 建立專案 | 群組旁的 + |
| 刪除專案 | 專案旁的 × → 二次確認 |
| 刪除資料夾 | 資料夾旁的 × → 二次確認 |
| 看到所有文件 | 不受 Viewers 限制 |

### 預設文件類型

| 類型 | 用途 |
|------|------|
| SRS | Software Requirements Specification |
| SDS | Software Design Specification |
| SPEC | 功能規格書 |
| PM-Doc | 產品需求文件 |
| Others | 其他 |

## 13. 資料庫設計

### 7 張資料表

```
docGroups          — 群組（最高層）
docProjects        — 專案
docCategories      — 資料夾分類（支援巢狀）
docTypes           — 文件類型（Admin 自訂）
docDocuments       — 文件主體
docVersions        — 版本快照（自動產生）

Join Tables（自動建立）：
  docDocumentViewers      (documentId, userId)
  docDocumentEditors      (documentId, userId)
  docDocumentSubscribers  (documentId, userId)
```
