# DocHub Plugin 使用手冊

> **DocHub** 是一套內建於 NocoBase 的企業文件管理系統，由 AI 全端工程師從零開始設計與實作。  
> 支援 Markdown 撰寫、**表單範本**、權限管理、版本歷史、全文搜尋、文件鎖定、稽核日誌、站內通知、Git 雙向同步（GitHub / GitLab），以及三層組織架構。

---

## 功能導覽

| 功能 | 解決的問題 |
|------|-----------|
| Markdown 編輯器（分割預覽） | 所有知識庫系統的基礎 |
| 三層組織架構（群組/專案/資料夾） | 對應組織結構：業主/案件/文件類型 |
| 建立專案自動產生 6 個資料夾 | 免手動建立常用分類，開箱即用 |
| 版本歷史 + 逐行 Diff | 文件被改壞找不回來；誰改了什麼不透明 |
| 全文搜尋（PostgreSQL GIN 索引） | 案件多了之後找文件靠捲動，效率極差 |
| **表單範本系統** | 上版單、申請單等結構化表單，告別醜陋的 Markdown 填表 |
| 文件鎖定（Lock / Unlock） | 重要文件防止被意外修改或刪除 |
| 稽核日誌（Audit Log） | 追蹤所有文件操作，符合合規需求 |
| 站內信通知（WebSocket 即時） | 協作時不知道文件被更新，各自改各自的 |
| Viewers / Editors / Subscribers | 文件不應讓所有人都能改，但相關人要能看 |
| GitHub / GitLab 雙向同步 | 文件同時在 Git 和系統，手動複製容易出版本落差 |
| GitLab Webhook 自動同步 | Git merge 後文件不自動更新，靠人記得手動拉 |
| 拖曳排序 + 欄寬調整 | 文件順序有業務意義，標題被截斷看不到 |

## 1. 系統簡介

DocHub 作為 NocoBase 自訂 Plugin 運行，完全整合在現有後台，不需要額外部署任何服務。

```
NocoBase v2.0.32 (Docker)
└── @nocobase/plugin-doc-hub
    ├── 前端 (AMD/UMD React，無 JSX)
    │   ├── ListPage            — 文件列表、搜尋、篩選
    │   ├── ViewPage            — 閱讀模式（Markdown 渲染 / 表單卡片）
    │   ├── EditPage            — Markdown 編輯器（分割預覽）
    │   ├── TemplateFillPage    — 表單填寫（新建 + 編輯）
    │   ├── TemplateListPage    — 範本管理（Admin）
    │   └── VersionPage         — 版本歷史 + 逐行 Diff
    └── 後端 (Node.js + Sequelize)
        ├── 自訂 Action Handlers（lock / unlock / webhookReceive / search）
        ├── 範本 CRUD Handlers（docTemplates:list/get/create/update/destroy）
        ├── afterCreate Hook（自動建 v1 版本 + 建立專案自動建 6 資料夾）
        ├── afterUpdate Hook（自動建版本快照、發站內通知）
        ├── beforeDestroy Hook（鎖定文件禁止刪除）
        └── PostgreSQL 資料庫（9 張表，含 docAuditLogs + docTemplates）
```

**技術亮點：**
- 純 `React.createElement` 撰寫，不需要 JSX 編譯，直接 deploy dist 即生效
- 完整 AMD/UMD 格式，符合 NocoBase 的 RequireJS 載入機制
- Markdown 渲染用本地 serve 的 `marked.min.js`，不依賴 CDN
- 支援 GitHub 與 GitLab（含自簽憑證）雙向同步

## 2. 快速上手

### 登入系統

1. 開啟瀏覽器，前往 `http://localhost:13000`
2. 輸入帳號 `admin@nocobase.com`、密碼 `admin123`
3. 點擊「登入」

### 進入 DocHub

登入後，點擊左側 sidebar 的 **DocHub** 連結，進入文件列表頁面。

### 介面概覽

```
┌─────────────────────────────────────────────────────────────────┐
│  Doc Hub                                          [新增文件]     │

```

> 操作欄圖示：✏ 編輯　📋 版本歷史　⇄ 移動文件　🗑 刪除　🔒 鎖定標示

## 3. 組織架構（群組 / 專案 / 資料夾）

DocHub 支援三層結構，讓大型團隊可以清楚分類管理文件。

```
群組（Group）          — 最高層，例：「專案」「共用」「部門文件」
  └── 專案（Project）  — 對應實際業務或 Git repo
       └── 資料夾（Category）  — 文件分類，支援無限層級巢狀
            └── 文件（Document）
```

### 建立群組（Admin 限定）

1. 在 sidebar「群組」標題右側點擊 **+**
2. 輸入群組名稱（例：`專案`、`共用`）
3. 點「建立」

### 建立專案（Admin 限定）

1. 在任一群組右側點擊 **+**
2. 輸入專案名稱
3. 點「建立」

> **建立專案後系統自動產生 6 個預設資料夾：**

| 資料夾 | 用途 |
|--------|------|
| SRS | Software Requirements Specification（需求規格） |
| SDS | Software Design Specification（設計規格） |
| SPEC | 功能規格書 |
| PM-Doc | 產品需求文件 |
| 會議記錄 | 會議紀錄與決議 |
| Others | 其他文件 |

這 6 個資料夾與系統內建的文件類型對齊，新增文件時可直接選擇，無需手動建立。

> 只有 Admin（NocoBase root / admin 角色）才能建立、刪除群組和專案。

### 刪除專案（Admin 限定）

1. 將滑鼠移到專案名稱上，右側出現 **×** 按鈕
2. 點擊 × → 彈出二次確認視窗
3. 確認視窗顯示紅色警告，必須點「確認刪除」才會執行

> **注意**：刪除專案不會自動刪除底下的文件，文件只是失去所屬專案。

### 建立資料夾

- **專案右側的 `+`** → 在該專案根目錄建立資料夾
- **資料夾右側的 `+`** → 在該資料夾底下建立子資料夾

建立時 Modal 會顯示「建立在『XXX』底下」提示確認位置。

> **注意：同一層不允許建立相同名稱的資料夾。**（系統會即時提示錯誤）

### 重新命名資料夾

**雙擊** sidebar 上的資料夾名稱 → 進入 inline 編輯模式：
- 輸入新名稱，按 **Enter** 確認 / **Escape** 取消 / 點其他地方也會儲存
- 若同層已有相同名稱，系統會提示錯誤並取消改名

### Sidebar 展開 / 折疊

- 點擊 ▸ / ▾ 展開或折疊子資料夾
- 點擊**資料夾名稱** → 右側只顯示該分類下的文件
- 點擊**專案名稱** → 顯示整個專案的文件
- 點擊「群組」右側的 **━━** 按鈕 → 一次縮起所有資料夾
- 同層資料夾可拖曳調整排序

## 4. 文件管理

### 新增文件

點擊右上角「**新增文件**」或列表中的 ✏ 編輯按鈕進入編輯頁。

#### 必填欄位

| 欄位 | 說明 | 備註 |
|------|------|------|
| **標題** | 文件名稱 | 同資料夾內必須唯一 |
| **所屬資料夾** | 文件歸屬位置 | 從 sidebar 進入新增會自動帶入 |
| **文件類型** | SRS / SDS / SPEC / PM-Doc / Others | 影響列表 Tab 篩選 |

#### 兩種建立模式

**模式 A：手動撰寫**

1. 點「新增文件」
2. 填寫文件標題
3. 選擇文件類型與狀態
4. 在左側 Markdown 編輯區撰寫，右側即時預覽
5. 點「儲存」或「發布」

**模式 B：從 Git 匯入**

1. 點「新增文件」
2. 在頁面頂部 Git 同步區填入：Repo、檔案路徑、分支
3. 點「從 Git 拉取內容」→ 自動填入文件內容
4. 補上標題後儲存

### 編輯文件（Markdown 編輯器）

```
┌─────────────────────────────────────────────────────┐
│  ← 返回  * [文件標題（必填）]    [儲存] [發布] [同步Git] │
├─────────────────────────────────────────────────────┤
│  所屬專案: [公司知識庫 ▼]  所屬資料夾: [Starters ▼]  │
│  文件類型: [SPEC ▼]        狀態: [Draft ▼]           │
├──────────────────────┬──────────────────────────────┤
│ B  I  U  H1 H2 H3   │                              │
│ — • 1. 🔗 </>       │                              │
│  [Markdown 輸入區]   │    [即時預覽]                │
└──────────────────────┴──────────────────────────────┘
```

**Markdown 工具列：**

| 按鈕 | 說明 |
|------|------|
| **B** | 粗體（`**text**`）|
| *I* | 斜體（`_text_`）|
| <u>U</u> | 底線（`<u>text</u>`）|
| H1 / H2 / H3 | 標題（`# ## ###`）|
| • | 無序列表 |
| 1. | 有序列表 |
| 🔗 | 超連結 |
| `</>` | 程式碼區塊 |

選取文字再點按鈕 → 自動包住選取的文字。

### 發布 / 草稿切換

| 動作 | 說明 |
|------|------|
| **儲存** | 存為 Draft，只有自己與有權限的人看得到 |
| **發布** | 狀態改為 Published，可同步到 Git |

儲存或發布時會彈出「版本備註」輸入框（選填），填寫後版本歷史會顯示說明文字。

### 版本歷史（Diff 比較）

在文件列表點擊 📋 按鈕 → 進入版本歷史頁。

- **建立文件時自動產生 v1 版本**，每次儲存/發布都自動快照
- 左側顯示版本列表（版本號、時間、編輯者、備註）
- 右側顯示逐行 Diff（綠色 `+` 為新增、紅色 `-` 為刪除）

### 移動文件

1. 在文件列表點擊 **⇄ 移動** 按鈕
2. 在 Modal 中選擇目標專案
3. 再選擇目標資料夾（可選）
4. 點「確認移動」

### 刪除文件

1. 在文件列表點擊 **🗑 刪除** 按鈕
2. 彈出二次確認視窗，確認文件名稱
3. 點「確認刪除」執行（此操作**不可復原**）

> **注意**：鎖定中的文件無法刪除，即使是管理員也一樣。必須先解鎖才能刪除。

## 5. 表單範本系統

DocHub 支援三種文件建立方式，透過「新增文件」按鈕的選擇對話框進入：

| 方式 | 說明 |
|------|------|
| ✍️ 自由撰寫 | Markdown 格式，彈性最高，適合技術文件、會議記錄等 |
| 📋 使用範本 | 選擇預設範本，填寫結構化表單，適合上版單、申請單、SOP 等 |
| 🔄 Git 同步 | 從 GitHub / GitLab 倉庫拉取 Markdown 文件 |

---

### 5.1 使用範本建立文件

**步驟：**

1. 在文件列表頁點擊右上角「**新增文件**」
2. 選擇「**📋 使用範本**」
3. 從範本清單中選擇對應範本（如「上版單」）
4. 輸入文件標題
5. 填寫表單欄位（必填欄位標有紅色 `*`）
6. 點擊「**儲存**」→ 跳轉至文件閱讀頁

**文件列表中的範本文件會顯示 `📋 範本` 標籤**，方便識別。

---

### 5.2 編輯範本文件

在文件閱讀頁點擊「**編輯**」按鈕，若是範本文件會自動跳轉至**表單填寫頁**（而非 Markdown 編輯器）。

表單資料以卡片 Grid 格式顯示，整齊呈現所有欄位及其值。

---

### 5.3 範本管理（Admin）

**入口：** 左側 Sidebar 底部「**📋 範本管理**」連結（僅 Admin 可見）

**功能：**
- 查看所有範本（可依專案篩選）
- 建立新範本
- 編輯現有範本
- 刪除範本（軟刪除，不影響已填寫的文件）

---

### 5.4 建立範本（範本建構器）

點擊「**建立範本**」或「**編輯**」按鈕開啟範本建構器對話框。

#### 基本資訊

| 欄位 | 說明 |
|------|------|
| 範本名稱 | 必填，例如：上版單、需求申請單 |
| 說明 | 選填，描述此範本的用途 |
| 預設資料夾 | 使用此範本建立的文件預設放置的資料夾 |

#### 欄位定義 — 視覺模式

拖曳 `⠿` 圖示可對欄位進行排序。每個欄位包含：

| 設定項 | 說明 |
|--------|------|
| 標籤（顯示名稱） | 用戶看到的欄位名稱，例如：「版本號」 |
| 系統名稱 | 自動由標籤轉換（可手動修改），例如：`version` |
| 欄位類型 | 見下表 |
| 必填 | 勾選後儲存時會驗證 |
| 預設值 | 選填，文字/日期類型適用 |

**支援的欄位類型：**

| 類型 | 說明 | 範例 |
|------|------|------|
| `文字（單行）` | 一般文字輸入 | 版本號、負責人姓名 |
| `文字（多行）` | 多行文字區塊 | 變更說明、影響範圍 |
| `日期` | 日期選擇器 | 上版日期、申請日期 |
| `單選` | 下拉選單，只能選一個 | 環境（dev/staging/prod）|
| `多選` | 下拉選單，可選多個 | 影響模組 |
| `用戶` | 系統用戶選擇器 | 負責人、審核人 |

**單選 / 多選選項格式：**
```
dev=dev
測試=staging
正式=prod
```
每行一個選項，格式為 `顯示文字=value`（若不需要不同 value，直接寫顯示文字即可）。

#### 欄位定義 — JSON 模式

點擊「**切換 JSON 模式**」可直接編輯 JSON，方便批量設定。切換回視覺模式時會自動解析。

```json
[
  {
    "id": "f_abc123",
    "type": "text",
    "label": "版本號",
    "name": "version",
    "required": true,
    "defaultValue": ""
  },
  {
    "id": "f_def456",
    "type": "select",
    "label": "上版環境",
    "name": "environment",
    "required": true,
    "defaultValue": "staging",
    "options": [
      {"label": "開發", "value": "dev"},
      {"label": "測試", "value": "staging"},
      {"label": "正式", "value": "prod"}
    ]
  }
]
```

#### 列表摘要欄位

勾選要在文件列表中顯示的欄位（最多 4 個），讓用戶無需點進文件就能看到關鍵資訊。

---

### 5.5 範本文件的資料格式

範本文件的 `content` 欄位以特殊格式儲存，與 Markdown 文件共存於同一張表：

```
TEMPLATE_FORM_V1
{"templateId":3,"templateName":"上版單","templateVersion":1,"data":{"version":"1.0","applyDate":"2026-04-15","environment":"prod","changeItems":"更新認證模組"}}
```

系統自動識別前綴 `TEMPLATE_FORM_V1` 來區分文件類型，對用戶完全透明。

## 6. 文件鎖定

### 什麼是鎖定？

文件鎖定（Lock）是一種**防寫保護機制**，用來保護重要文件不被意外修改或刪除。

鎖定後的影響：
- 非管理員無法編輯文件內容
- **任何人（包括管理員）都無法刪除**鎖定中的文件
- 文件列表的該文件列會顯示 🔒 鎖定標示
- 只有管理員可以鎖定或解鎖

> **注意**：鎖定保護的是文件內容與刪除操作。管理員仍可解鎖文件後進行後續操作。

### 如何鎖定文件（管理員操作）

1. 在文件列表找到要鎖定的文件
2. 點擊該文件列的「**⋯**」更多操作按鈕
3. 選擇「**鎖定文件**」
4. 彈出確認 Modal，顯示以下警告：

   > **此操作將保護文件不被修改或刪除。非管理員將無法編輯，任何人將無法刪除此文件，直到管理員解鎖為止。**

5. 確認無誤後點「**確認鎖定**」
6. 文件列出現 🔒 標示，鎖定完成

> 點「取消」則放棄操作，文件狀態不變。

### 如何解鎖文件（管理員操作）

1. 在文件列表找到鎖定中（🔒）的文件
2. 點擊「**⋯**」更多操作按鈕
3. 選擇「**解鎖文件**」
4. 確認 Modal 中點「**確認解鎖**」
5. 🔒 標示消失，文件恢復正常狀態

### 鎖定後的行為對照表

| 操作 | 管理員 | 一般用戶（Editors） | 未授權用戶 |
|------|--------|---------------------|-----------|
| 閱讀文件 | 可以 | 可以 | 看不到（受 Viewers 限制）|
| 編輯文件 | 可以 | **不可以（403）** | 不可以 |
| 刪除文件 | **不可以（403）** | 不可以 | 不可以 |
| 解鎖文件 | 可以 | 不可以 | 不可以 |

> **重要**：鎖定文件時，刪除操作對所有人（包括管理員）均被拒絕，這是刻意設計，確保鎖定具有真正的保護效果。

## 8. Git 雙向同步

每篇文件可以個別綁定一個 Git 檔案路徑，支援 **GitHub** 和 **GitLab**（含公司內網自簽憑證）。

> **Demo Repo**：`https://10.1.2.191/wezoomtek/wez-spring-boot-starters`  
> 已包含 `docs/` 目錄和多份 `.md` 檔案，可直接用來體驗 Git 雙向同步。

### 前置設定（管理員操作）

在使用 Git 同步前，管理員需先在伺服器設定 Token：

#### 產生 GitLab Personal Access Token

1. 登入 GitLab（`https://10.1.2.191`）
2. 右上角頭像 → **Edit profile** → **Access Tokens**
3. 建立新 Token，Scopes 勾選：
   - ✅ `read_repository` — 允許讀取（拉取）
   - ✅ `write_repository` — 允許寫入（推送）
4. 複製 Token（只顯示一次）

#### 填入伺服器 .env

```env
DOCHUB_GITLAB_HOST=10.1.2.191
DOCHUB_GITLAB_TOKEN=glpat-xxxxxxxxxxxxxxxxxxxx
```

重啟服務：`docker-compose restart app`

---

### 支援的 Git 平台

| 平台 | Repo 格式 | 範例 |
|------|---------|------|
| GitHub | `owner/repo` | `yanchen184/my-project` |
| GitLab（IP） | `10.1.2.191/namespace/project` | `10.1.2.191/wezoomtek/backend` |
| GitLab（含 https）| `https://...` | 兩種格式皆支援 |

### 在文件上綁定 Git

在 EditPage 頂部（Admin 才可見）的 Git 同步列填入：

| 欄位 | 說明 | 範例 |
|------|------|------|
| **Repo** | Git repo 路徑 | `wezoomtek/wez-spring-boot-starters` |
| **檔案路徑** | repo 內的相對路徑 | `docs/quick-start.md` |
| **分支** | 要同步的分支 | `main` 或 `master` |

### Git → DocHub：從 Git 拉取最新內容

**方式 A：手動拉取**
1. 進入文件的 EditPage
2. 在頂部 Git 同步列點「**從 Git 拉取最新**」
3. 成功後 Editor 內容會更新為 Git 上的最新版

**方式 B：Webhook 自動觸發（推薦）**

設定後，每次 `git push` 會自動把最新 `.md` 同步到 DocHub：

1. 進入 GitLab 專案 → **Settings** → **Webhooks**
2. URL 填入：
   ```
   http://your-server-ip:13000/api/docDocuments:webhookReceive
   ```
3. **Trigger** 勾選 `Push events`
4. 若使用自簽憑證，取消勾選 `Enable SSL verification`
5. 點 **Add webhook** → **Test** → 確認回傳 `200 OK`

### DocHub → Git：推送到 Git

1. 文件狀態必須為**已發布**（草稿無法推送）
2. 在文件列表找到該文件，點 **⋯** → **同步 Git**
3. 系統將 DocHub 的 Markdown 內容推送到 GitLab 對應路徑，並自動建立 commit

> 搭配 **SourceTree** 可直觀地看到 DocHub 推送的 commit 記錄：  
> commit message 格式為 `docs: update {文件標題} via DocHub`

### 同步失敗通知

當同步失敗（Token 過期、repo 路徑錯誤、網路不通等），系統會：
1. 將文件的同步狀態設為 `Failed`
2. 在列表頁顯示紅色失敗標示
3. 發送**站內信**通知所有管理員

### 同步狀態說明

| 狀態 | 含義 |
|------|------|
| 空白 | 未設定 Git 連結 |
| `Synced ✓`（綠） | 最後一次同步成功 |
| `Failed ✗`（紅） | 同步失敗，請確認 Token 或 repo 設定 |
| `Pending`（灰） | 設定了但尚未同步過 |

## 9. 權限管理

### 資料夾層級權限

每個資料夾可以設定哪些用戶有讀取或編輯權限。設定方式：

1. 在 sidebar 右鍵點擊資料夾（或點擊設定圖示）
2. 選擇「資料夾權限設定」
3. 新增 **Viewer**（只能讀取）或 **Editor**（可讀取並編輯）用戶

| 角色 | 可以看資料夾下的文件 | 可以新增/編輯文件 |
|------|---------------------|-----------------|
| Viewer | 是 | 否 |
| Editor | 是 | 是 |

### 文件層級權限

每篇文件在 EditPage 的 Permissions 區塊可以獨立設定：

| 角色 | 說明 |
|------|------|
| **Viewers** | 可以看到這篇文件（不在列表就不顯示）|
| **Editors** | 可以進入編輯模式修改（自動包含讀取）|

| 身份 | 可看到的文件 |
|------|------------|

> **重要**：沒有設定任何 Viewers 的文件，一般用戶在列表看不到，也無法進入。

## 10. 稽核日誌

稽核日誌（Audit Log）是**管理員專用**功能，記錄所有文件操作歷程，用於合規審查與問題追蹤。

### 如何開啟稽核日誌

1. 進入 DocHub 文件列表頁
2. 在 sidebar 或頂部選單找到「**稽核日誌**」連結（僅管理員可見）
3. 點擊後開啟 Modal，顯示操作記錄表格

> **注意**：一般用戶存取 `/api/docAuditLogs` 會回傳 403，無法查閱稽核日誌。

### 記錄的操作類型

| 動作 | 觸發時機 |
|------|---------|
| **建立** | 新增文件時 |
| **編輯** | 修改文件內容或屬性時 |
| **發布** | 文件狀態從 Draft 改為 Published 時 |
| **刪除** | 刪除文件時 |
| **鎖定** | 管理員鎖定文件時 |
| **解鎖** | 管理員解鎖文件時 |
| **Git 同步** | 從 Git 拉取或推送到 Git 時 |

### 稽核日誌欄位說明

| 欄位 | 說明 |
|------|------|
| 時間 | 操作發生的時間戳記 |
| 動作 | 操作類型（建立、編輯、刪除、鎖定…） |
| 文件標題 | 被操作的文件名稱 |
| 操作者 | 執行操作的用戶帳號 |
| 備註 | 額外說明（版本備註、失敗原因等）|

## 11. 站內信通知（鈴鐺圖示）

文件的 **Subscribers** 在文件被修改時，自動收到 NocoBase 站內信通知。通知透過 WebSocket 即時推送，無需重新整理頁面。

### 通知內容範例

```
標題：文件更新：後端部署 SOP
內容：Tom 更新了文件《後端部署 SOP》（修正 API 路徑錯誤）
```

### 什麼時候會收到通知？

以下事件會觸發站內信通知：

| 事件 | 通知對象 |
|------|---------|
| 文件被修改（含內容 / 標題）| 文件的所有 Subscribers |
| 文件被發布 | 文件的所有 Subscribers |
| Git 同步失敗 | 文件建立者 + 管理員 |

### 如何訂閱文件通知？

1. 進入文件的 EditPage
2. 在 **Permissions** 區塊找到「Subscribers」欄位
3. 輸入要訂閱的用戶名稱並新增
4. 儲存文件後設定生效

### 通知規則

- 有填版本備註 → 通知裡附上備註內容
- 通知出現在 NocoBase 右上角鈴鐺圖示，顯示紅點提示（WebSocket 即時推送）
- 編輯者本人不會收到自己所做修改的通知（防止自我通知刷屏）
- Git 同步失敗時，同時通知文件建立者與管理員，提醒盡快修復同步設定

### 查看通知

點擊頁面右上角的**鈴鐺圖示** → 展開通知清單，點擊任一通知可跳轉至對應文件。

## 12. 常見問題 FAQ

**Q：文件建立後，一般用戶在列表看不到怎麼辦？**

A：需要在 EditPage 的 Permissions 區塊，將該用戶加入 Viewers、Editors 或 Subscribers 任一清單。沒有設定的文件，一般用戶完全看不到。

---

**Q：建立專案後，為什麼 sidebar 自動出現 6 個資料夾？**

A：這是設計好的預設結構（SRS、SDS、SPEC、PM-Doc、會議記錄、Others），對應常見文件類型，省去手動建立步驟。可以繼續在底下新增子資料夾。

---

**Q：鎖定文件後我連刪除都不行？**

A：正確。鎖定設計上對所有人（包括管理員）禁止刪除，確保保護效果。請先解鎖文件，再執行刪除。

---

**Q：Git 同步失敗，怎麼排查？**

A：依序檢查：
1. Repo 路徑格式是否正確（GitHub: `owner/repo`；GitLab: IP 或 domain 開頭）
2. Git Token 是否過期（在環境變數中更新 `GITHUB_TOKEN` 或 `GITLAB_TOKEN`）
3. 檔案路徑是否存在於對應分支
4. 若為 GitLab 自簽憑證，確認 Node.js 設定跳過 SSL 驗證

---

**Q：Webhook 自動同步設定後沒反應？**

A：確認以下項目：
1. Webhook URL 是否為**公開可存取**的 HTTPS 地址（ngrok 或固定 IP）
2. GitLab 的 SSL verification 是否已**取消勾選**（ngrok 需要）
3. ngrok 免費版每次重啟 URL 會改變，需重新設定 Webhook URL
4. 確認 Trigger 為 Push events

---

**Q：版本歷史保留幾個版本？**

A：目前無上限，每次儲存/發布都自動建立版本快照。若文件頻繁修改導致版本過多，可由管理員直接透過 NocoBase 後台清理 `docVersions` 表。

---

**Q：稽核日誌可以匯出嗎？**

A：目前版本尚未支援匯出功能。管理員可透過 NocoBase 後台直接查詢 `docAuditLogs` 資料表，或使用資料庫管理工具（如 pgAdmin）匯出 CSV。

---

**Q：如何新增或修改文件類型（如新增「API Doc」類型）？**

A：管理員可透過 NocoBase 後台 → 資料管理 → `docTypes` 表，直接新增記錄。新增後重新整理頁面即可在 EditPage 的文件類型下拉選單看到新選項。

---

**Q：範本文件可以轉回 Markdown 嗎？**

A：目前不支援自動轉換。範本文件的內容以特殊格式（`TEMPLATE_FORM_V1`）儲存，若需要 Markdown 格式，建議新建一份自由撰寫的文件。

## 12. UX 快捷操作

### 鍵盤快捷鍵

#### 編輯頁（EditPage）

| 快捷鍵 | 功能 |
|--------|------|
| `⌘S` / `Ctrl+S` | 儲存文件（Draft） |
| `⌘⇧S` / `Ctrl+Shift+S` | 發布文件（Published） |
| `⌘B` / `Ctrl+B` | 粗體（包住選取文字） |
| `⌘I` / `Ctrl+I` | 斜體 |
| `⌘U` / `Ctrl+U` | 底線 |

#### 全域快捷鍵（任何頁面）

| 快捷鍵 | 功能 |
|--------|------|
| `⌘K` / `Ctrl+K` | 聚焦側邊欄搜尋框 |

### 自動儲存草稿

- 編輯頁有內容變更後，右上角出現 **「未儲存」** 紅點提示
- 每 **30 秒**自動儲存一次（已存在的文件 + 有標題才觸發）
- 自動儲存成功後顯示「已自動儲存 HH:MM:SS」
- 離開編輯頁時若有未儲存變更，瀏覽器會彈出警告

### 閱讀進度條

文件閱讀頁頂部有一條 **3px 藍綠漸層進度條**，隨滾動比例自動更新，方便長文件閱讀追蹤。

### 列印 / 存 PDF

在文件閱讀頁點擊工具列「**列印**」按鈕（或按 `⌘P`）：
- 觸發瀏覽器列印對話框
- 側邊欄、工具列等操作區塊自動隱藏，只保留文件內容
- 在列印對話框選擇「另存為 PDF」可匯出 PDF 檔

### 最近查看

- 側邊欄底部（管理員入口上方）顯示**最近查看的 5 篇文件**
- 每次開啟文件自動記錄，最多保留 8 筆
- 點擊即可快速跳轉

### 專案層級訂閱者

在專案的 🔐 權限設定 Modal 中，第三個選擇器「**📬 文件訂閱者**」：
- 設定後，該專案所有文件的修改通知都會發給這些用戶
- 與個別文件的 Subscribers 合併發送（去重複）
- 適合：希望追蹤整個專案所有文件動態的 PM / TL

---

**Q：刪除範本後，已用該範本建立的文件會怎樣？**

A：刪除範本是「軟刪除」（status 改為 archived），已填寫的文件資料不受影響，仍可正常閱讀。若範本定義找不到，ViewPage 會以通用 key-value 格式顯示表單資料。

---

**Q：範本欄位的系統名稱（name）有什麼限制？**

A：系統名稱由標籤自動產生（中文轉底線格式），也可手動修改。限制為英文、數字、底線組合，且同一範本內必須唯一。建議不要更改已有填寫資料的範本欄位名稱，否則舊資料的對應會遺失。
