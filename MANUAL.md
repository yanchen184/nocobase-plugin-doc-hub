# DocHub 使用手冊

> **DocHub** 是一套內建於公司後台的文件管理系統。用它撰寫、組織、搜尋、協作所有專案文件，也能和 Git 雙向同步。

---

## 這份手冊適合誰？

- 需要撰寫 / 閱讀 / 查找專案文件的所有同仁
- 負責文件分類與權限管理的專案負責人
- 想把文件和 Git repository 同步的工程師

---

## 功能一覽

| 功能 | 你可以用它做什麼 |
|------|----------------|
| Markdown 編輯器 | 寫文件時左邊輸入、右邊即時預覽 |
| 三層組織架構（群組/專案/資料夾） | 依照業主、案件、文件類型把文件分門別類 |

| 全文搜尋 | 同時搜尋標題和內容，命中處會黃底高亮 |
| 表單範本系統 | 上版單、申請單等格式化表單，填空式建立不怕漏欄 |
| 文件鎖定 | 重要文件上鎖避免被意外修改或刪除 |
| 稽核日誌 | 查看誰在什麼時間做了哪些操作 |
| 站內信通知 | 訂閱文件 → 有更新時 NocoBase 右上鈴鐺會響 |
| 權限管理（Viewer / Editor / Subscriber） | 設定誰能看、誰能改、誰會收到通知 |
| GitHub / GitLab 雙向同步 | 文件變動可推到 Git，Git merge 也能回流到文件 |
| 拖曳排序 + 欄寬調整 | 依你習慣的順序與欄位寬度排版 |

## 1. 系統簡介

DocHub 是整合在公司後台（NocoBase）內的文件管理系統，登入後台後直接從左側選單進入，不需要另外安裝任何工具。

它能幫你：
- **集中管理文件**：所有專案文件放在同一個地方，不再散落各處
- **結構化分類**：三層組織（群組 → 專案 → 資料夾）對應實際業務
- **追蹤變更**：每次修改都有版本紀錄，誰改了什麼一目瞭然
- **協同作業**：分配權限、訂閱通知，讓對的人看到對的文件
- **和 Git 同步**：重要文件可雙向同步到 GitHub / GitLab

## 2. 快速上手

### 登入系統

1. 開啟瀏覽器，前往公司提供的後台網址
2. 輸入公司配發給你的帳號與密碼
3. 點擊「登入」

> 若忘記帳號或密碼，請聯絡系統管理員。

### 進入 DocHub

登入後，點擊左側選單的 **DocHub**，進入文件列表頁面。

### 介面概覽

進入 DocHub 後預設看到的文件列表頁，由左側組織樹（群組/專案/資料夾）＋頂部類型 Tab＋中間文件表格組成。

> **操作欄圖示**：✏ 編輯　📋 版本歷史　⇄ 移動文件　🗑 刪除　🔒 鎖定 / 🔓 解鎖　🔄 Git 同步

## 3. 組織架構（群組 / 專案 / 資料夾）

DocHub 採用三層結構來管理文件。**第一層固定為兩個群組：「專案」與「共用」**，第二層之後才是各式各樣的內容。

```
群組（Group）                     — 最高層，固定兩個：「專案」「共用」
  ├── 專案                        — 放各式各樣實際案件、產品、repo
  │   ├── wez-spring-boot-starters
  │   │   └── 資料夾 / 文件
  │   └── 客戶 A 官網改版
  │       └── 資料夾 / 文件
  └── 共用                        — 放跨專案的共用規範、模板、SOP
      ├── 01_技術規範              （後端 / 前端 / 資料庫 / DevOps…）
      │   └── 資料夾 / 文件
      └── 02_公司開發規範
          └── 資料夾 / 文件
```

**使用時機：**

| 群組 | 放什麼 | 範例 |
|------|--------|------|
| **專案** | 每個實際案件、產品、repo 各開一個專案；名稱直接用實際案件名 | `wez-spring-boot-starters`、`客戶 A 官網改版` |
| **共用** | 跨專案共用的規範、模板、SOP、會議記錄；建議加上 `01_`、`02_` 數字前綴維持排序 | `01_技術規範`、`02_公司開發規範` |

> **命名慣例：** 只有「共用」群組底下的項目建議加上 `01_`、`02_` 前綴，方便維持分類排序；「專案」群組直接用案件真實名稱即可。

### 建立群組（只在首次部署時需要）

一般情況下，系統已預設建立好「專案」與「共用」兩個群組，使用者不需要自己建立新的群組。如需調整，管理員可依下列步驟操作：

1. 在 sidebar「群組」標題右側點擊 **+**
2. 輸入群組名稱
3. 點「建立」

### 建立專案

1. 在「專案」或「共用」群組右側點擊 **+**
2. 輸入專案名稱（實際案件名稱，或共用文件集合的名稱）
3. 點「建立」

> **建立專案後系統自動產生 6 個預設資料夾：**

| 資料夾 | 用途 |
|--------|------|
| SRS | Software Requirements Specification（需求規格） |
| SDS | Software Design Specification（設計規格） |
| SPEC | 功能規格書 |
| PM-Doc | 產品需求文件 |
| 會議記錄 | 會議紀錄與決議 |
| Others | 其他文件 |

這 6 個資料夾與系統內建的文件類型對齊，新增文件時可直接選擇，無需手動建立。

> 所有登入使用者都可以建立、編輯、刪除群組和專案（依 NocoBase ACL 預設設定為 public）。若要限制，請在 NocoBase「角色與權限」中調整 `docGroups` / `docProjects` 資源的 Action 權限。

### 刪除專案

1. 將滑鼠移到專案名稱上，右側出現 **×** 按鈕
2. 點擊 × → 彈出二次確認視窗
3. 確認視窗顯示紅色警告，必須點「確認刪除」才會執行

> **注意**：刪除專案不會自動刪除底下的文件，文件只是失去所屬專案。

### 建立資料夾

- **專案右側的 `+`** → 在該專案根目錄建立資料夾
- **資料夾右側的 `+`** → 在該資料夾底下建立子資料夾

建立時 Modal 會顯示「建立在『XXX』底下」提示確認位置。

> **注意：同一層不允許建立相同名稱的資料夾。**（系統會即時提示錯誤）

### 重新命名資料夾

**雙擊** sidebar 上的資料夾名稱 → 進入 inline 編輯模式：
- 輸入新名稱，按 **Enter** 確認 / **Escape** 取消 / 點其他地方也會儲存
- 若同層已有相同名稱，系統會提示錯誤並取消改名

### Sidebar 展開 / 折疊

- 點擊 ▸ / ▾ 展開或折疊子資料夾
- 點擊**資料夾名稱** → 右側只顯示該分類下的文件
- 點擊**專案名稱** → 顯示整個專案的文件
- 點擊「群組」右側的 **━━** 按鈕 → 一次縮起所有資料夾
- 同層資料夾可拖曳調整排序

## 4. 文件管理

### 新增文件

點擊右上角「**新增文件**」或列表中的 ✏ 編輯按鈕進入編輯頁。

#### 必填欄位

| 欄位 | 說明 | 備註 |
|------|------|------|
| **標題** | 文件名稱 | 同資料夾內必須唯一 |
| **所屬資料夾** | 文件歸屬位置 | 從 sidebar 進入新增會自動帶入 |
| **文件類型** | SRS / SDS / SPEC / PM-Doc / Others | 影響列表 Tab 篩選 |

#### 兩種建立模式

**模式 A：手動撰寫**

1. 點「新增文件」
2. 填寫文件標題
3. 選擇文件類型與狀態
4. 在左側 Markdown 編輯區撰寫，右側即時預覽
5. 點「儲存」或「發布」

**模式 B：從 Git 匯入**

1. 點「新增文件」
2. 在頁面頂部 Git 同步區填入：Repo、檔案路徑、分支
3. 點「從 Git 拉取內容」→ 自動填入文件內容
4. 補上標題後儲存

### 編輯文件（Markdown 編輯器）

編輯頁採左右分割版面：左側 Markdown 輸入區、右側即時預覽；頂部為文件標題輸入＋儲存/發布/同步 Git 按鈕；上方 metadata 列提供所屬專案、資料夾、類型、狀態下拉。（畫面參見本章最後「文件編輯頁（分割預覽模式）」截圖）

**編輯器工具列：**

| 按鈕 | 功能 |
|------|------|
| **B** | 粗體 |
| *I* | 斜體 |
| <u>U</u> | 底線 |
| H1 / H2 / H3 | 標題（由大到小共三級） |
| • | 項目符號清單 |
| 1. | 編號清單 |
| 🔗 | 插入超連結 |
| `</>` | 插入程式碼區塊 |

先用滑鼠選取文字，再點上方按鈕，系統會自動套用樣式。

### 發布 / 草稿切換

| 動作 | 說明 |
|------|------|
| **儲存** | 存為 Draft，只有自己與有權限的人看得到 |
| **發布** | 狀態改為 Published，可同步到 Git |

儲存或發布時會彈出「版本備註」輸入框（選填），填寫後版本歷史會顯示說明文字。

### 版本歷史（Diff 比較）

在文件列表點擊 📋 按鈕 → 進入版本歷史頁。

- **建立文件時自動產生 v1 版本**，每次儲存/發布都自動快照
- 左側顯示版本列表（版本號、時間、編輯者、備註）
- 右側顯示逐行 Diff（綠色 `+` 為新增、紅色 `-` 為刪除）

### 移動文件

1. 在文件列表點擊 **⇄ 移動** 按鈕
2. 在 Modal 中選擇目標專案
3. 再選擇目標資料夾（可選）
4. 點「確認移動」

### 刪除文件

1. 在文件列表點擊 **🗑 刪除** 按鈕
2. 彈出二次確認視窗，確認文件名稱
3. 點「確認刪除」執行（此操作**不可復原**）

> **注意**：鎖定中的文件無法刪除，即使是管理員也一樣。必須先解鎖才能刪除。

## 5. 表單範本系統

DocHub 支援三種文件建立方式，透過「新增文件」按鈕的選擇對話框進入：

| 方式 | 說明 |
|------|------|
| ✍️ 自由撰寫 | Markdown 格式，彈性最高，適合技術文件、會議記錄等 |
| 📋 使用範本 | 選擇預設範本，填寫結構化表單，適合上版單、申請單、SOP 等 |
| 🔄 Git 同步 | 從 GitHub / GitLab 倉庫拉取 Markdown 文件 |

---

### 5.1 使用範本建立文件

**步驟：**

1. 在文件列表頁點擊右上角「**新增文件**」
2. 選擇「**📋 使用範本**」
3. 從範本清單中選擇對應範本（如「上版單」）
4. 輸入文件標題
5. 填寫表單欄位（必填欄位標有紅色 `*`）
6. 點擊「**儲存**」→ 跳轉至文件閱讀頁

**文件列表中的範本文件會顯示 `📋 範本` 標籤**，方便識別。

---

### 5.2 編輯範本文件

在文件閱讀頁點擊「**編輯**」按鈕，若是範本文件會自動跳轉至**表單填寫頁**（而非 Markdown 編輯器）。

表單資料以卡片 Grid 格式顯示，整齊呈現所有欄位及其值。

---

### 5.3 範本管理（Admin）

**入口：** 左側 Sidebar 底部「**📋 範本管理**」連結（僅 Admin 可見）

**功能：**
- 查看所有範本（可依專案篩選）
- 建立新範本
- 編輯現有範本
- 刪除範本（軟刪除，不影響已填寫的文件）

---

### 5.4 建立範本（範本建構器）

點擊「**建立範本**」或「**編輯**」按鈕開啟範本建構器對話框。

#### 基本資訊

| 欄位 | 說明 |
|------|------|
| 範本名稱 | 必填，例如：上版單、需求申請單 |
| 說明 | 選填，描述此範本的用途 |
| 預設資料夾 | 使用此範本建立的文件預設放置的資料夾 |

#### 欄位定義 — 視覺模式

拖曳 `⠿` 圖示可對欄位進行排序。每個欄位包含：

| 設定項 | 說明 |
|--------|------|
| 標籤（顯示名稱） | 用戶看到的欄位名稱，例如：「版本號」 |
| 系統名稱 | 自動由標籤轉換（可手動修改），例如：`version` |
| 欄位類型 | 見下表 |
| 必填 | 勾選後儲存時會驗證 |
| 預設值 | 選填，文字/日期類型適用 |

**支援的欄位類型：**

| 類型 | 說明 | 範例 |
|------|------|------|
| `文字（單行）` | 一般文字輸入 | 版本號、負責人姓名 |
| `文字（多行）` | 多行文字區塊 | 變更說明、影響範圍 |
| `日期` | 日期選擇器 | 上版日期、申請日期 |
| `單選` | 下拉選單，只能選一個 | 環境（dev/staging/prod）|
| `多選` | 下拉選單，可選多個 | 影響模組 |
| `用戶` | 系統用戶選擇器 | 負責人、審核人 |

**單選 / 多選選項格式：**
```
dev=dev
測試=staging
正式=prod
```
每行一個選項，格式為 `顯示文字=value`（若不需要不同 value，直接寫顯示文字即可）。

#### 欄位定義 — JSON 模式（進階）

若你熟悉 JSON 格式，可點擊「**切換 JSON 模式**」直接編輯欄位定義，方便批次設定。切換回視覺模式時系統會自動解析。

> 大部分使用者不需要使用此模式，視覺模式已足以應付所有場景。

#### 列表摘要欄位

勾選要在文件列表中顯示的欄位（最多 4 個），讓使用者不用點進文件就能看到關鍵資訊，例如「版本號」「上版環境」。

## 6. 文件鎖定

### 什麼是鎖定？

文件鎖定（Lock）是一種**防寫保護機制**，用來保護重要文件不被意外修改或刪除。

鎖定後的影響：
- 非管理員無法編輯文件內容
- **任何人（包括管理員）都無法刪除**鎖定中的文件
- 文件列表的該文件列會顯示 🔒 鎖定標示
- 只有管理員可以鎖定或解鎖

> **注意**：鎖定保護的是文件內容與刪除操作。管理員仍可解鎖文件後進行後續操作。

### 如何鎖定文件（管理員操作）

1. 在文件列表找到要鎖定的文件
2. 點擊該文件列的「**⋯**」更多操作按鈕
3. 選擇「**鎖定文件**」
4. 彈出確認 Modal，顯示以下警告：

   > **此操作將保護文件不被修改或刪除。非管理員將無法編輯，任何人將無法刪除此文件，直到管理員解鎖為止。**

5. 確認無誤後點「**確認鎖定**」
6. 文件列出現 🔒 標示，鎖定完成

> 點「取消」則放棄操作，文件狀態不變。

### 如何解鎖文件（管理員操作）

1. 在文件列表找到鎖定中（🔒）的文件
2. 點擊「**⋯**」更多操作按鈕
3. 選擇「**解鎖文件**」
4. 確認 Modal 中點「**確認解鎖**」
5. 🔒 標示消失，文件恢復正常狀態

### 鎖定後的行為對照表

| 操作 | 管理員 | 一般用戶（Editors） | 未授權用戶 |
|------|--------|---------------------|-----------|
| 閱讀文件 | 可以 | 可以 | 看不到（受 Viewers 限制）|
| 編輯文件 | 可以 | **不可以（403）** | 不可以 |
| 刪除文件 | **不可以（403）** | 不可以 | 不可以 |
| 解鎖文件 | 可以 | 不可以 | 不可以 |

> **重要**：鎖定文件時，刪除操作對所有人（包括管理員）均被拒絕，這是刻意設計，確保鎖定具有真正的保護效果。

## 7. 全文搜尋

在 sidebar 頂部的搜尋框輸入關鍵字，400ms 自動觸發搜尋。

### 搜尋範圍

- 文件**標題**
- 文件**全文內容**（不分大小寫，使用資料庫全文索引加速）

### 搜尋結果顯示

- 標題命中 → 標題文字**黃底高亮**
- 內容命中 → 每筆文件顯示最多 **3 個命中片段**（前後各 60 字），黃色左邊線標示
- 片段裡的關鍵字同樣黃底高亮

### 組合篩選

搜尋可以和以下條件同時使用：
- Sidebar 的**專案**篩選
- Sidebar 的**資料夾**篩選
- 頂部的**文件類型** Tab（全部 / SRS / SDS / SPEC / PM-Doc / Others）

## 8. Git 雙向同步

每篇文件可以個別綁定一個 Git 檔案路徑，支援 **GitHub** 和 **GitLab**（含公司內網自簽憑證）。

> **Demo Repo**：`https://10.1.2.191/wezoomtek/wez-spring-boot-starters`  
> 已包含 `docs/` 目錄和多份 `.md` 檔案，可直接用來體驗 Git 雙向同步。

> **給一般使用者**：如果你只是要「使用」Git 同步功能，看下一節「在文件上綁定 Git」即可。本節的「前置設定」屬於管理員一次性的部署工作。

### 支援的 Git 平台

| 平台 | Repo 格式 | 範例 |
|------|---------|------|
| GitHub | `owner/repo` | `yanchen184/my-project` |
| GitLab（IP） | `10.1.2.191/namespace/project` | `10.1.2.191/wezoomtek/backend` |
| GitLab（含 https）| `https://...` | 兩種格式皆支援 |

### 在文件上綁定 Git

在 EditPage 頂部（Admin 才可見）的 Git 同步列填入：

| 欄位 | 說明 | 範例 |
|------|------|------|
| **Repo** | Git repo 路徑 | `wezoomtek/wez-spring-boot-starters` |
| **檔案路徑** | repo 內的相對路徑 | `docs/quick-start.md` |
| **分支** | 要同步的分支 | `main` 或 `master` |

### Git → DocHub：從 Git 拉取最新內容

**方式 A：手動拉取**
1. 進入文件的編輯頁面
2. 在頂部 Git 同步列點「**從 Git 拉取最新**」
3. 成功後編輯區內容會更新為 Git 上的最新版

**方式 B：自動同步（由管理員設定 Webhook 後啟用）**

設定完成後，每次有人 `git push`，Git 平台會主動把變更通知 DocHub，文件內容自動保持最新——使用者不需要手動拉取。

> 詳細的 Webhook 設定步驟請參考章末「管理員附錄：Git 同步初次設定」。

### DocHub → Git：推送到 Git

1. 文件狀態必須為**已發布**（草稿無法推送）
2. 在文件列表找到該文件，點 **⋯** → **同步 Git**
3. 系統將 DocHub 的 Markdown 內容推送到 GitLab 對應路徑，並自動建立 commit

> 搭配 **SourceTree** 可直觀地看到 DocHub 推送的 commit 記錄：  
> commit message 格式為 `docs: update {文件標題} via DocHub`

### 同步失敗通知

當同步失敗（Token 過期、repo 路徑錯誤、網路不通等），系統會：
1. 將文件的同步狀態設為 `Failed`
2. 在列表頁顯示紅色失敗標示
3. 發送**站內信**通知所有管理員

### 同步狀態說明

| 狀態 | 含義 |
|------|------|
| 空白 | 未設定 Git 連結 |
| `Synced ✓`（綠） | 最後一次同步成功 |
| `Failed ✗`（紅） | 同步失敗，請確認 Token 或 repo 設定 |
| `Pending`（灰） | 設定了但尚未同步過 |

## 9. 權限管理

DocHub 採用**三層權限**架構，由上而下繼承：專案層 → 資料夾層 → 文件層。

### 9.1 專案層級權限

每個專案可設定 Viewer / Editor / Subscriber 三種角色。設定後，權限**自動套用到專案下所有資料夾和文件**（除非資料夾有自訂設定）。

**操作方式**：點選專案列右側的 🔐 按鈕。

| 角色 | 說明 |
|------|------|
| **Viewer** | 可查看此專案下所有文件 |
| **Editor** | 可查看並編輯此專案下所有文件 |
| **Subscriber** | 任何文件有更新時自動收到站內信通知 |

---

### 9.2 資料夾層級權限

每個資料夾預設**繼承所屬專案的設定**。Admin 可以對特定資料夾設定自訂權限，覆蓋專案設定。

**操作方式**：hover 資料夾列，點選 🔓（繼承）或 🔐（自訂中）圖示。

**權限模式說明**：

| 模式 | 圖示 | 說明 |
|------|------|------|
| 繼承專案設定（預設）| 🔓 | 直接套用所屬專案的 Viewer / Editor |
| 自訂此資料夾的權限 | 🔐 | 獨立設定，不受專案權限影響 |

> 切換回「繼承」時，資料夾自訂的成員清單會自動清空。

---

### 9.3 文件層級權限

每篇文件在 EditPage 的 Permissions 區塊可以獨立設定（更細粒度的控制）：

| 角色 | 說明 |
|------|------|
| **Viewers** | 可以看到這篇文件（不在列表就不顯示）|
| **Editors** | 可以進入編輯模式修改（自動包含讀取）|
| **Subscribers** | 文件每次修改時自動收到站內通知（自動包含讀取）|

---

### 9.4 存取規則

| 身份 | 可看到的文件 |
|------|------------|
| **Admin** | 所有文件，不受任何權限限制 |
| **文件建立者** | 自己建立的所有文件，自動擁有完整權限 |
| **一般用戶** | 文件層、資料夾層（自訂）、或專案層（繼承）任一有授權即可見 |

**權限查找優先順序**：

```
1. 文件層有授權？→ 可見
2. 資料夾有自訂權限且 user 在清單？→ 可見
3. 資料夾繼承專案，且 user 在專案 Viewer/Editor？→ 可見
4. 以上都沒有 → 不可見
```

> **重要**：沒有任何授權的文件，一般用戶在列表完全看不到，也無法進入。

## 10. 稽核日誌

稽核日誌（Audit Log）是**管理員專用**功能，記錄所有文件操作歷程，用於合規審查與問題追蹤。

### 如何開啟稽核日誌

1. 進入 DocHub 文件列表頁
2. 在 sidebar 或頂部選單找到「**稽核日誌**」連結（僅管理員可見）
3. 點擊後開啟 Modal，顯示操作記錄表格

> **注意**：一般用戶存取 `/api/docAuditLogs` 會回傳 403，無法查閱稽核日誌。

### 記錄的操作類型

| 動作 | 觸發時機 |
|------|---------|
| **建立** | 新增文件時 |
| **編輯** | 修改文件內容或屬性時 |
| **發布** | 文件狀態從 Draft 改為 Published 時 |
| **刪除** | 刪除文件時 |
| **鎖定** | 管理員鎖定文件時 |
| **解鎖** | 管理員解鎖文件時 |
| **Git 同步** | 從 Git 拉取或推送到 Git 時 |

### 稽核日誌欄位說明

| 欄位 | 說明 |
|------|------|
| 時間 | 操作發生的時間戳記 |
| 動作 | 操作類型（建立、編輯、刪除、鎖定…） |
| 文件標題 | 被操作的文件名稱 |
| 操作者 | 執行操作的用戶帳號 |
| 備註 | 額外說明（版本備註、失敗原因等）|

## 11. 站內信通知（鈴鐺圖示）

文件的 **Subscribers** 在文件被修改時，自動收到 NocoBase 站內信通知。通知透過 WebSocket 即時推送，無需重新整理頁面。

### 通知內容範例

```
標題：文件更新：後端部署 SOP
內容：Tom 更新了文件《後端部署 SOP》（修正 API 路徑錯誤）
```

### 什麼時候會收到通知？

以下事件會觸發站內信通知：

| 事件 | 通知對象 |
|------|---------|
| 文件被修改（含內容 / 標題）| 文件的所有 Subscribers |
| 文件被發布 | 文件的所有 Subscribers |
| Git 同步失敗 | 文件建立者 + 管理員 |

### 如何訂閱文件通知？

1. 進入該文件的編輯頁面
2. 在「權限」區塊找到 **Subscribers（訂閱者）** 欄位
3. 輸入要訂閱通知的使用者並新增
4. 儲存文件即生效

### 通知規則

- 有填版本備註 → 通知內文會附上備註內容
- 通知出現在右上角鈴鐺圖示，即時顯示紅色未讀數字
- 編輯者本人不會收到自己改動的通知
- Git 同步失敗時，會同時通知文件建立者與管理員

### 查看通知

點擊頁面右上角的**鈴鐺圖示** → 展開通知清單，點擊任一通知可跳轉到對應文件。

## 12. 附件管理（PDF / Word / Excel）

DocHub 支援在文件中嵌入 **PDF / Word / Excel / PowerPoint / 壓縮檔** 等附件。

- **PDF**：直接在閱讀頁以 iframe 內嵌預覽，不用離開 DocHub
- **Word / Excel / 其他檔案**：以美化的下載卡片顯示，點一下即可下載

支援檔案類型：`.pdf`、`.doc`、`.docx`、`.xls`、`.xlsx`、`.ppt`、`.pptx`、`.csv`、`.txt`、`.zip`、`.rar`、`.7z`、`.md`（單檔上限 **50 MB**）。

### 如何上傳附件？

在編輯頁工具列點擊 **📎 附件** 按鈕，選取本機檔案即可：

- 上傳完成後，系統自動把對應的 Markdown 語法插入游標位置
- PDF 使用 `!pdf[檔名](url)` 語法（閱讀頁渲染為內嵌預覽）
- 其他檔案使用 `[📎 檔名](url)` 語法（閱讀頁渲染為下載卡片）

也可以 **直接拖曳檔案** 到編輯框，或 **從剪貼簿貼上**，系統會自動上傳並插入 Markdown。

### Markdown 語法

```markdown
# PDF 內嵌（自動渲染為 iframe）
!pdf[季度報告 Q1](/storage/uploads/doc-files/xxxx_sample.pdf)

# 其他附件（自動渲染為下載卡片）
[📎 會議記錄_20260424.docx](/storage/uploads/doc-files/xxxx_sample.docx)
[📎 預算表_2026.xlsx](/storage/uploads/doc-files/yyyy_budget.xlsx)
```

### 閱讀頁呈現

- **PDF**：頂部顯示檔名與「下載 ↗」連結，下方嵌入 600px 高的 PDF 檢視器（可滾動、放大、列印）
- **其他附件**：顯示圖示（Word=📝 / Excel=📊 / PowerPoint=📽 / Zip=🗄 / 其他=📎）+ 檔名，點擊即下載

### 常見問題

**Q：PDF 內嵌看不到內容？**
A：確認瀏覽器允許內嵌 PDF（部分瀏覽器需要內建 PDF Viewer 擴充）。若仍無法顯示，請點右上角「下載 ↗」以下載檢視。

**Q：可以上傳圖片嗎？**
A：可以。編輯頁工具列另有 **🖼 圖片** 按鈕專門處理圖片上傳，會以 `![alt](url)` 語法插入。

**Q：上傳失敗出現「檔案過大」？**
A：單檔上限 50 MB。大檔案請考慮改用雲端儲存並貼連結，或拆分成多個檔案。

**Q：附件可以刪除嗎？**
A：在編輯頁直接刪除對應的 Markdown 語法即可（儲存後該附件不再顯示）。實際檔案會由系統定期清理。

## 13. 標籤管理（Tags）

### 標籤是什麼？

標籤（Tag）是「橫跨專案／資料夾」的彈性分類方式。適合用來標註跨專案的共通屬性，例如 `urgent`、`security`、`migration`、`proposal` 等。

- **全域**：所有專案共用同一套標籤（不是每個專案各自一套）。
- **自由建立**：任何使用者在編輯文件時，直接打字 Enter 就會自動建立新標籤。
- **自動配色**：第一次建立時系統依名稱 hash 自動挑選顏色，管理員可在標籤管理頁修改。
- **可搜尋**：在任何頁面點一下標籤，即可跳到「帶有此標籤」的文件清單。

### 為文件加入標籤（編輯頁）

1. 進入文件的編輯頁。
2. 在頁面頂部的屬性列找到「**標籤**」欄位。
3. 直接輸入標籤名稱，按 Enter 即可加入。輸入已存在的名稱會自動顯示下拉候選。
4. 儲存文件後生效。

單一文件不限制標籤數量，但建議 3–5 個以內以保持清晰。

### 透過標籤找到相關文件

- **點擊任意文件卡片 / 詳情頁的標籤**：切到「只看此標籤」的列表。
- **側邊欄「🏷 標籤」區塊**：顯示使用次數最多的 Top 10 標籤；點擊即篩選。
- **URL 參數 `?tags=A,B`**：可在列表頁直接帶多個標籤（含任一者就顯示）。

> 篩選邏輯為 **「OR（包含任一）」**：`?tags=urgent,security` 會列出「帶 urgent **或** security」的文件。

### 標籤管理頁（`/admin/doc-hub/tags`）

僅管理員可存取。進入方式：

- 側邊欄「🏷 標籤」區塊右側「管理…」按鈕。
- 或直接造訪 `/admin/doc-hub/tags`。

可操作：

| 功能 | 說明 |
|------|------|
| **編輯** | 重新命名、自訂 HEX 色碼 |
| **合併** | 把「A」的所有文件改標成「B」，然後刪除「A」。適合整理雷同命名的標籤（如 `bug` 與 `bugs`） |
| **刪除** | 移除標籤。若該標籤仍被某些文件使用，刪除會同時從所有文件中取下此標籤 |

### 權限模型

- 標籤本身不做權限控制；任何使用者都可以建立、套用。
- **可見性由文件權限繼承**：若使用者對某文件沒有 view 權限，在標籤篩選結果中也不會看到該文件。因此篩選「urgent」看到的是「自己可見且帶有 urgent 的文件」，不會洩露他人私有文件。
- `docTags:update` / `docTags:merge` / `docTags:destroy` 僅限管理員。

### 常見問題

**Q：可以為不同專案建立同名但不同顏色的標籤嗎？**
A：不行。標籤是全域的，`name` 為唯一鍵。若需要差異化，請用不同名稱（如 `ops-urgent`、`dev-urgent`）。

**Q：誤刪標籤怎麼辦？**
A：直接重新建立同名標籤即可——但原本掛在這個標籤的文件關聯會遺失，需要重新標註。刪除前請確認「使用次數」欄位，避免誤傷。

**Q：合併會弄丟東西嗎？**
A：不會。合併是「將 A 的所有關聯改成指向 B，再刪除 A」；文件本身不會被動到，只有標籤關聯會轉移。

- 點擊即可快速跳轉

### 專案層級訂閱者

在專案的 🔐 權限設定 Modal 中，第三個選擇器「**📬 文件訂閱者**」：
- 設定後，該專案所有文件的修改通知都會發給這些用戶
- 與個別文件的 Subscribers 合併發送（去重複）
- 適合：希望追蹤整個專案所有文件動態的 PM / TL

## 15. 常見問題 FAQ

**Q：文件建立後，一般用戶在列表看不到怎麼辦？**

A：需要在 EditPage 的 Permissions 區塊，將該用戶加入 Viewers、Editors 或 Subscribers 任一清單。沒有設定的文件，一般用戶完全看不到。

---

**Q：建立專案後，為什麼 sidebar 自動出現 6 個資料夾？**

A：這是設計好的預設結構（SRS、SDS、SPEC、PM-Doc、會議記錄、Others），對應常見文件類型，省去手動建立步驟。可以繼續在底下新增子資料夾。

---

**Q：鎖定文件後我連刪除都不行？**

A：正確。鎖定設計上對所有人（包括管理員）禁止刪除，確保保護效果。請先解鎖文件，再執行刪除。

---

**Q：Git 同步失敗，怎麼排查？**

A：依序檢查：
1. Repo 路徑格式是否正確（GitHub: `owner/repo`；GitLab: IP 或 domain 開頭）
2. Git Token 是否過期（在環境變數中更新 `GITHUB_TOKEN` 或 `GITLAB_TOKEN`）
3. 檔案路徑是否存在於對應分支
4. 若為 GitLab 自簽憑證，確認 Node.js 設定跳過 SSL 驗證

---

**Q：Git 上已經 push 了，為什麼 DocHub 沒自動更新？**

A：自動同步需由管理員設定 Webhook 才會啟用。若你已經 push 了但文件沒更新：
1. 先在文件的編輯頁點「**從 Git 拉取最新**」手動觸發
2. 若手動也沒反應，請聯絡管理員確認 Webhook 設定（詳見章末「管理員附錄」）

---

**Q：版本歷史保留幾個版本？**

A：目前無上限，每次儲存/發布都自動建立版本快照。若文件頻繁修改導致版本過多，可由管理員直接透過 NocoBase 後台清理 `docVersions` 表。

---

**Q：稽核日誌可以匯出嗎？**

A：管理員可透過 NocoBase 後台直接查詢 `docAuditLogs` 資料表，或使用資料庫管理工具（如 pgAdmin）匯出 CSV。

---

**Q：如何新增或修改文件類型（如新增「API Doc」類型）？**

A：管理員可透過 NocoBase 後台 → 資料管理 → `docTypes` 表，直接新增記錄。新增後重新整理頁面即可在 EditPage 的文件類型下拉選單看到新選項。

---

**Q：範本文件可以轉回 Markdown 嗎？**

A：範本文件的內容以特殊格式（`TEMPLATE_FORM_V1`）儲存，若需要 Markdown 格式，建議另外新建一份「自由撰寫」類型的文件。

---

**Q：刪除範本後，已用該範本建立的文件會怎樣？**

A：刪除範本是「軟刪除」（status 改為 archived），已填寫的文件資料不受影響，仍可正常閱讀。若範本定義找不到，ViewPage 會以通用 key-value 格式顯示表單資料。

---

**Q：範本欄位的系統名稱（name）有什麼限制？**

A：系統名稱由標籤自動產生（中文轉底線格式），也可手動修改。限制為英文、數字、底線組合，且同一範本內必須唯一。建議不要更改已有填寫資料的範本欄位名稱，否則舊資料的對應會遺失。

## 管理員附錄：Git 同步初次設定

本節只需由系統管理員執行一次，完成後所有使用者即可使用第 8 章的 Git 同步功能。

### 步驟 1：產生 GitLab Personal Access Token

1. 登入公司 GitLab
2. 右上角頭像 → **Edit profile** → **Access Tokens**
3. 建立新 Token，Scopes 勾選：
   - ✅ `read_repository` — 允許讀取（拉取）
   - ✅ `write_repository` — 允許寫入（推送）
4. 複製 Token（只顯示一次，請妥善保管）

### 步驟 2：將 Token 填入伺服器環境變數

在伺服器的 `.env` 檔填入：

```env
DOCHUB_GITLAB_HOST=<公司 GitLab host>
DOCHUB_GITLAB_TOKEN=<剛才產生的 Token>
```

然後重啟服務：`docker-compose restart app`

### 步驟 3：設定 Webhook（啟用自動同步）

對每個需要自動同步的 GitLab 專案：

1. 進入 GitLab 專案 → **Settings** → **Webhooks**
2. URL 填入：`http://<伺服器>/api/docDocuments:webhookReceive`
3. **Trigger** 勾選 `Push events`
4. 若使用自簽憑證，取消勾選 `Enable SSL verification`
5. 點 **Add webhook** → **Test** → 確認回傳 `200 OK`

完成後，GitLab 上的 `git push` 會自動把變更同步到 DocHub，使用者無需手動操作。
